# 給美術：情報卡 UI 規格（原：證物卡 / 線索載體）

> 目的：讓「線索」不只是敘述，而是玩家看得見、可核對的「載體化資訊」。
>
> 重點先講清楚：**本案線索多為「人際/情緒/界線/權力動態」→ 不必做成圖片證物**。  
> 我們先做 **Text-first 情報卡**：用固定 UI 皮膚承載「對話摘錄/條款摘錄/時間線摘錄/通知摘錄」，內容可以後續由 AI 產出並填入。
>
> 工程會用 `clueId` 對應 `evidenceType`（見 `../給工程/證物資料結構.md`）。

---

## 0) 最小可交付（MVP）

- **每條 CLUE 至少 1 張「情報卡」**（同一套 UI 皮膚重用，不要求逐條繪製）
- **情報卡由 4 種載體皮膚組成**（先做這四種就夠）：
  1. **手機通知/訊息（Notification）**
  2. **群聊截圖（Chat Screenshot）**
  3. **交付時間線（Timeline / Log）**
  4. **會議文件（Document / PIP）**

### 0.1 無圖模式（建議預設）

- **不需要任何「具象圖片證物」**（例如真的畫一張群聊截圖圖檔）
- 只要做：
  - 4 種載體的 **UI 皮膚**
  - 高亮樣式（紅框/黃底/底線）
  - 字體層級與間距
- **內容來源**：工程填「摘錄文字」（可由 AI 生成 / 模板拼裝），美術不需要參與每條線索內容生產

## 1) 共用元件：EvidenceCard（情報卡）

### 1.1 版面（建議）

- **卡片頭**：證物類型標籤 + 時間（例：2019/09/10 18:32）
- **卡片主體**：依 evidenceType 呈現
- **卡片腳**：來源（例：會議室 / 群聊 / 交付紀錄）+ 小字「可核對」

### 1.2 視覺語氣

- **乾淨、冷、像真實系統 UI**（不要花哨）
- **重點高亮**：紅框/黃底/底線（用來對齊企劃中「這句就是線索」）
- **避免太戲劇化**：不是恐怖片，是制度壓迫

## 2) 載體元件規格

### A) Notification（手機通知/訊息）

**用途**：CLUE-04/10 這類「發送訊息=通報」的瞬間。

- **必要素材**：
  - 手機狀態列（電量/訊號/時間）
  - App icon（可用 generic chat icon）
- **可變欄位（由工程填）**：
  - 發訊者/群組名（可匿名化）
  - 內容（短句）
  - 時間戳
- **美術提示**：
  - 通知「亮起又被按掉」可以用兩張狀態（未讀/消失）做切換

### B) Chat Screenshot（群聊截圖）

**用途**：CLUE-12（風向）或任何「斷章取義」場景。

- **必要素材**：
  - 群聊 header（群名/成員數）
  - 氣泡（左右）
  - 時間分隔線（可選）
- **可變欄位**：
  - 3–6 則訊息（短句）
  - 一則被紅框/黃底標記的「斷章取義句」
- **美術提示**：
  - 讓「轉傳」看起來像轉傳（forward 小字/引用框）

### C) Timeline / Log（交付時間線/紀錄頁）

**用途**：CLUE-06（知情）、M05/M09 的「可核對事實」打法。

- **必要素材**：
  - 垂直時間線（點/線）
  - 每筆 log 卡（時間/動作/附件）
  - 右上角「導出/下載」小 icon（可假）
- **可變欄位**：
  - 4–6 筆記錄（提交/確認/轉交/改期）
  - 其中 1–2 筆被標記（就是線索所在）

### D) Document / PIP（會議文件 / 改善計畫）

**用途**：CLUE-11（HR 程序壓迫、主管切割）這類文件證據。

- **必要素材**：
  - A4 文件版面（標題/章節/表格）
  - 「請簽名」區塊
  - 日期/編號欄位
- **可變欄位**：
  - 標題（例：改善計畫）
  - 3–5 行重點條款（可假文）
  - 被圈起來的關鍵句（例如「已記錄」）

## 4) 什麼時候才「需要做圖」？

只有以下情況，才值得投入「圖片證物」：

- **玩家需要辨識非文字訊號**：例如「通知亮起又被按掉」這種節奏（仍可用 UI 切狀態取代）
- **需要超高真實感/分享感**：例如結局前的「終局審判」要能截圖分享
- **你要做『可驗證』的推理玩法**：例如玩家要比對截圖中的時間戳、附件檔名（那就做 1–2 張代表性圖即可）
